Models & Inference

LLMLarge Language Model

A neural network trained on massive text datasets that can generate, summarize, classify, and transform text. As an AI engineer you treat it as a black-box API — you send tokens in, tokens come back.

Examples: Claude (Anthropic), GPT-4 (OpenAI), Gemini (Google), Llama (Meta). You rarely choose one for capability alone — cost, latency, and context window size drive the decision.

LLM
Token

The atomic unit an LLM processes. Roughly 0.75 English words, or 4 characters. Models don't see words or characters — they see tokens.

Why it matters: you pay per token and hit limits per token. "1M token context" ≈ 750,000 words ≈ a small novel.

"Hello, world!" → ["Hello", ",", " world", "!"] # 4 tokens
LLM
Context Window

The maximum number of tokens an LLM can process in a single call — input + output combined. If you exceed it, the call fails.

Claude Sonnet 4.6: 200k tokens input, 8k output. GPT-4o: 128k. Larger windows cost more per call. RAG exists partly to work around this limit.

LLM
Model Tier

Providers offer models at different capability/cost/speed tiers. Choose by matching task complexity to tier, not by always picking the best model.

Anthropic tiers (2026): Haiku (fast/cheap) → Sonnet (balanced) → Opus (most capable). Route simple classification to Haiku, complex reasoning to Sonnet/Opus.

LLM

Prompting

System Prompt

Instructions sent before the conversation that set the model's role, output format, and behavioral constraints. The model treats these as standing orders.

system="You are an invoice parser. Return only valid JSON. Never add commentary."
LLM
Structured Output

Getting an LLM to return data in a predictable schema (usually JSON) instead of free-form text. Essential for any production system that needs to parse the response.

Enforce this with Pydantic validation on the receiving end. Never trust unvalidated LLM JSON in production.

{"vendor": "Acme", "amount": 1250.00, "date": "2024-03-15"}
LLM
Temperature

A sampling parameter (0.0–1.0) controlling output randomness. Lower = more deterministic, higher = more creative/varied.

For extraction and classification tasks: use 0. For creative tasks: 0.7–1.0. Default (unset) is usually around 1.0.

LLM

Data & Retrieval

Embedding

A fixed-size numeric vector (e.g. 1536 floats) that represents the semantic meaning of a piece of text. Texts with similar meaning have similar vectors (high cosine similarity).

You don't read embeddings directly — you compare them mathematically. Generated by embedding models, not chat models.

Data
Vector Database

A database optimized for storing embeddings and finding the most similar ones to a query vector (approximate nearest-neighbor search).

Options: pgvector (Postgres extension, simplest for existing Postgres users), Pinecone (managed), Chroma (local dev). Start with pgvector unless you have a reason not to.

Data
Chunking

Splitting source documents into smaller pieces before embedding, since embedding models have input limits and smaller chunks retrieve more precisely.

Chunk size is a hyperparameter. Typical: 500–1000 tokens with 100-token overlap. Too small = loss of context. Too large = imprecise retrieval.

Data
RAGRetrieval-Augmented Generation

Pattern where relevant documents are retrieved from a database and injected into the prompt before calling the LLM. Lets the model answer questions about data it wasn't trained on.

query → embed query → find similar chunks → inject into prompt → LLM answers with citations

Agents & Workflows

Tool CallingFunction Calling

Capability where you declare functions (tools) the LLM can invoke. The model returns a structured tool call request; your code executes it and feeds results back.

The model doesn't execute code — it requests execution. Your application is in control.

tools=[{"name": "get_weather", "input_schema": {"city": "string"}}] # Model returns: {"name": "get_weather", "input": {"city": "Paris"}}
LLM
Agent

An LLM system that uses tool calling in a loop: the model decides what action to take, your code executes it, the result feeds back to the model — repeat until done.

Most production "agents" are bounded state machines, not fully autonomous. Unbounded agents are unreliable. Add human approval gates for consequential actions.

LLM
State Machine

A workflow where the LLM system progresses through explicit states with defined transitions. More predictable and debuggable than open-ended agents.

LangGraph implements this pattern. Prefer state machines over open loops when reliability matters.

Infra
Guardrails

Checks on LLM inputs and outputs that catch harmful, off-topic, or malformed responses before they reach the user or downstream systems.

Can be rule-based (regex, blocklists) or model-based (a second LLM call to evaluate the first). Model-based guardrails add latency and cost.

Infra

Production & Ops

Observability

Logging, tracing, and monitoring of LLM calls in production. Captures: inputs, outputs, latency, token count, cost, errors, and model version per request.

Tools: LangSmith, Helicone, Braintrust. Blind spots kill production LLM systems — you can't tune what you can't see.

Infra
EvaluationEvals

Systematic measurement of LLM output quality. Methods: human review, rule-based checks (regex, exact match), model-based (LLM-as-judge scoring a response).

Ship evals before shipping features. "Does this work?" is a question evals answer at scale, not vibes.

Eval
LLM-as-Judge

Using a capable LLM to score or classify outputs from another LLM call. The judge scores responses on a rubric (1–5, pass/fail, etc.).

Surprisingly effective. Bias exists (models prefer their own outputs) — use a different model as judge than the one being evaluated.

Eval
Latency

Time from sending an API request to receiving the complete response. For user-facing features, target <2s for most interactions; streaming hides latency.

Main knobs: model tier (smaller = faster), max_tokens (cap output size), streaming (start showing output immediately).

Infra
Cost-per-Run

Total token cost of executing one workflow: (input tokens × input price) + (output tokens × output price). Track this metric per workflow from day one.

A workflow that costs $0.02 at 100 runs/day costs $730/year. At 10k runs/day: $73,000/year. Know your numbers before you scale.

Infra
Idempotency

Designing operations so that running them multiple times produces the same result as running once. Critical for LLM pipelines with retries.

Use idempotency keys on API calls. Cache LLM responses where input is identical. Avoid double-processing in queue workers.
