Embeddings — Deep Dive

Key questions and answers from the lesson discussion. For quick refresh, not re-learning from scratch.

What embeddings are

What is an embedding?

A fixed-length list of floats (768–3072 numbers) that encodes the semantic meaning of text. Produced by passing text through a transformer model. The numbers are meaningless individually — only their relationships matter.

Texts with similar meaning → similar vectors (small angle between them). Different topics → vectors far apart.

How does "unpaid" match "overdue" when neither word appears in the other document?

Embedding models are trained on billions of documents. "Unpaid" and "overdue" appear in nearly identical surrounding contexts (payment reminders, accounting docs). The model pushes their vectors together because words that appear in the same contexts get similar vectors — not because anyone taught it they're synonyms.

This is why semantic search beats keyword search: it retrieves by concept, not spelling.

How does the model encode a whole sentence, not just individual words?

Transformer models use attention: every token looks at every other token before producing its representation. The word "overdue" in "Invoice is overdue" is influenced by "Invoice" and "pay" — so it ends up in a different part of the space than "overdue" in "Our meeting is long overdue".

The model outputs one final vector per input that's a compressed summary of the whole meaning — not a per-word average.

Cosine similarity — worked example

Using 2D vectors for clarity. Real embeddings have 768 dimensions but the formula is identical.

A = [1, 2]   → "invoice overdue Poland"
B = [2, 4]   → "unpaid invoice Warsaw"  (same direction, scaled)
C = [2, 1]   → "recipe for pierogi"     (different direction)
1
Dot product — sum(x×y for x,y in zip(a,b))
A·B = (1×2) + (2×4) = 2 + 8 = 10
A·C = (1×2) + (2×1) = 2 + 2 = 4
2
Norms — sqrt(sum(x² for x in a))
|A| = √(1²+2²) = √5 ≈ 2.236
|B| = √(2²+4²) = √20 ≈ 4.472
|C| = √(2²+1²) = √5 ≈ 2.236
3
Divide — dot / (norm_a × norm_b)
cosine(A,B) = 10 / (2.236 × 4.472) = 10/10.0 = 1.0 ← identical direction
cosine(A,C) = 4 / (2.236 × 2.236) = 4/5.0 = 0.8 ← different direction
Why not just use dot product? Longer documents have larger magnitude vectors. Without dividing by the norms, a long document would dominate similarity scores even if it's less relevant. Normalising makes it a fair comparison — only direction counts.

Clusters and embedding space

What is a "cluster" — is it a physical region in the vector database?

Yes, exactly. Each embedding is a point in N-dimensional space. A cluster is a region of that space where similar documents happen to land — not because anyone drew a boundary, but because they emerged from training.

The geometry works the same as 2D: nearby points = similar meaning, distant points = different meaning. Cosine similarity measures the angle from the origin, which is equivalent to asking "are these points in the same region?"

Clusters are regions of N-dimensional space. Boundaries are fuzzy — some documents sit between clusters.

Why 768 dimensions / attention heads

Why 768 dimensions specifically?

Design choice from BERT (2018): 12 attention heads × 64 dimensions each = 768. It worked well, got published, and became the default most models copied. Same reason HTTP uses port 80 — someone picked it, it stuck.

More dimensions = more capacity for nuance, but more memory and slower search. 768 hits an empirical sweet spot. Going to 10,000 dimensions would likely perform worse due to the curse of dimensionality (everything starts looking equally far apart).

What are attention heads?

Attention is the mechanism where each word "looks at" every other word to decide how much to borrow from them. A single head can only learn one type of relationship at a time, so 12 heads run in parallel — each learning to specialise in something different:

Head 1  → syntactic structure  (subject → verb → object)
Head 2  → financial entities   (Invoice, Beta Ltd cluster)
Head 3  → payment status words (overdue, paid, outstanding)
Head 4  → negation             ("not paid" vs "paid")
...     → emerged from training, not manually assigned

Each head works in a 64-dimensional subspace. After all 12 run in parallel, their outputs are concatenated back into 768 dimensions — 12 different "perspectives" stitched together.

What about domain-specialised models (e.g. coding)?

A specialised model is not a slice of a general model's space. It's a completely separately trained model — same 768 dimensions — but trained predominantly on domain data (GitHub, Stack Overflow, docs). The space is denser in that domain and sparse in others:

General model:    finance ████  cooking ████  code ████  medicine ████
Code model:       code ████████████  finance ██  cooking █

The attention heads learned finer distinctions within the domain (async def vs def, Python vs JavaScript) because that's what the training data rewarded.

Use specialised models when your documents are mostly in one domain and retrieval precision matters. They'll perform poorly outside that domain.

Images, text-to-image & multimodal models

How do images get embedded? Is it the same as text?

Same process, different input. Instead of splitting text into word tokens, an image is split into patches (small fixed-size squares, e.g. 14×14 pixels). Each patch becomes a token. Those patch tokens go through a transformer — each patch attends to every other patch — and you get one vector out.

text:  "invoice overdue"  → word tokens  → transformer → [768 floats]
image: photo.jpg          → patches      → transformer → [768 floats]

Same output shape. The architecture is identical once you've tokenised the input.

Image → patches → transformer → one 768-float vector. Same architecture as text, different tokenisation.

What is CLIP and why does it matter?

CLIP (OpenAI, 2021) trained a model on 400M image-text pairs from the web. It learned to embed images and text into the same vector space, so similar concepts land near each other regardless of modality. 

"a dog running on a beach"  →  [0.23, -0.81, ...]
photo of dog on beach       →  [0.21, -0.79, ...]  ← close
photo of an invoice         →  [-0.4,  0.12, ...]  ← far

This enables text→image search: embed the query text, find images with the nearest vectors. Also the foundation of text-to-image generation.

CLIP maps text and images into the same space. Similar concepts cluster together regardless of modality.

How does text-to-image generation work? (Stable Diffusion, DALL-E)

These are generative models, not embedding models. The core idea is diffusion — gradually removing noise from a random image, guided at every step by a text vector.

  1. Text prompt → encode into a vector (CLIP-like)
  2. Start with a completely random noise image
  3. Run ~20–50 denoising steps — at each step, nudge the image to look more like what the text vector describes
  4. Coherent image emerges

The text vector acts as a compass: "make this noise look more like a dog on a beach." The model never "draws" — it denoises.

Diffusion: start with noise, denoise step-by-step. The text vector steers every step.

How do multimodal LLMs work? (GPT-4o, LLaVA, Claude)

These take both text and images as input in the same conversation. The trick: project image patches into the text token embedding space, then feed everything to a normal LLM.

  1. Image → split into patches → visual token embeddings (via vision encoder)
  2. Project visual tokens into text token space (learned linear layer)
  3. Concatenate: [visual tokens] + [text tokens]
  4. Feed into a standard LLM transformer — image is just more tokens

The model doesn't "see" an image — it sees a sequence of visual tokens mixed with text tokens, and attends across all of them.

Multimodal LLMs concatenate visual tokens and text tokens, then run a standard transformer. The image is not special — it's just tokens.

The through-line across all of this: everything reduces to "get inputs into a shared vector space, then run transformer attention over them." Text, images, audio (Whisper tokenises audio spectrograms the same way) — the architecture is the same once you've solved how to tokenise each modality. This is why multimodal AI is moving fast: adding a new modality is mostly a tokenisation problem, not an architecture redesign.
