RAG Evaluation

Cheat sheet for measuring RAG quality — faithfulness, relevance, and the LLM-as-judge pattern — Lesson 07

The Three Metrics

Faithfulness

Is the answer grounded in the retrieved context, or did the model make things up?

"Does the answer stay within what the context actually says?"

Low score → hallucination. The model went beyond its source.

Answer Relevance

Does the answer address what the user actually asked?

"Does the answer answer the question?"

Low score → off-topic answer. Model answered a different question.

Context Relevance

Did retrieval fetch chunks that are actually useful for answering the question?

"Are the retrieved chunks about the right thing?"

Low score → retrieval failure. Wrong chunks, wrong threshold, wrong embeddings.

Rule of thumb: Faithfulness catches generation failures. Context relevance catches retrieval failures. Answer relevance catches both — a useful final sanity check.

Diagnosing Failures

Symptom Which metric fails Likely cause Fix
Answer contradicts the document Faithfulness Model hallucinating beyond context Tighten system prompt: "answer ONLY from context"
Answer is correct but doesn't answer the question Answer Relevance Model answered adjacent question Rephrase user prompt; add few-shot examples
Answer confidently wrong, context is empty Context Relevance Similarity threshold too loose; wrong embedding model Raise threshold; check embedding model domain fit
Correct fact missing from answer Context Relevance Chunk boundary split the key fact Reduce chunk size or increase overlap
Irrelevant chunks retrieved for off-topic query Context Relevance Faithfulness Threshold too low; model hallucinates from weak context Raise distance threshold; add refusal path when context empty
All metrics fine but pass rate low All Test cases are too hard or judge threshold too strict Review test cases; calibrate threshold against human labels

LLM-as-Judge Pattern

question + context + answer
→
judge LLM call
→
{"score": 4, "reason": "..."}
→
pass / fail
def judge(prompt: str) -> tuple[int, str]: resp = client.chat.completions.create( model=CHAT_MODEL, messages=[{"role": "user", "content": prompt}], response_format={"type": "json_object"}, # force JSON output ) data = json.loads(resp.choices[0].message.content) return int(data["score"]), data["reason"] # The judge prompt must describe the JSON schema you want back: # "Return JSON: {"score": <1-5>, "reason": "<one sentence>"}"
Why it works

The judge sees the context and answer simultaneously and can compare them in a way you can't automate with string matching. It produces a numeric score you can track over time.

Works because LLMs are good at following a rubric — even a self-evaluation rubric — when the task is well-specified.

Limitations

Adds latency and cost — each test case requires extra model calls. The judge can be inconsistent across runs. A stronger judge model gives more reliable scores.

In production: run evals offline as a batch job, not per-user. Cache results. Use a stronger judge than the model you're evaluating when possible.

Faithfulness prompt
Rate how faithful this answer is to the context. 5 = fully supported, no hallucinations. 1 = contradicts or ignores context. Return JSON: {"score": <1-5>, "reason": "<one sentence>"} Context: {context} Answer: {answer}
Relevance prompt
Rate how well this answer addresses the question. 5 = directly and completely answers it. 1 = misses the point entirely. The answer should mention: {must_contain} Return JSON: {"score": <1-5>, "reason": "<one sentence>"} Question: {question} Answer: {answer}

Refusal casemust_contain = ""

When must_contain is empty, the correct answer is a refusal ("I don't have relevant information"). Set f_score = 5 — a refusal makes no claims, so it can't hallucinate. Let r_score judge whether the refusal was appropriate.

Pass threshold

A common default is PASS_THRESHOLD = 3 (out of 5). Raise to 4 for stricter evals, lower to 2 only when debugging. Track threshold alongside pass rate so results are comparable across runs.

What Metrics Enable

Safe refactoring

Change chunk size, similarity threshold, system prompt, or model — then re-run the eval. If pass rate drops, you have a regression. Without a metric, every change is a guess.

Stakeholder reporting

"Our RAG answers 92% of test questions correctly, up from 67% after tuning the threshold" is a sentence you can say in a meeting. "It feels better" is not.

Regression detection

Run the eval in CI before deploying. A drop in pass rate blocks the deploy. This is unit testing for LLM behaviour.
