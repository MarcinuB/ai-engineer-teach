AI Engineering Lesson 04

Embeddings & Semantic Search

Turn text into numbers that capture meaning — the foundation of RAG.

What is an embedding?

An embedding model takes any piece of text and outputs a fixed-length list of floats — typically 768 to 3072 numbers. The numbers are meaningless individually, but the list as a whole encodes the semantic meaning of the text.

Texts with similar meaning end up with similar vectors. Texts about different topics end up far apart. The diagram below shows a 2D projection — real embeddings have hundreds of dimensions, but the clustering behaviour is the same.

Similar texts cluster together. The query lands near overdue invoices — those get retrieved.

This is the core of RAG (Retrieval-Augmented Generation): embed a user query, compare it against embedded documents, retrieve the top matches, feed them to the LLM as context. You'll build the full pipeline in the next lesson.

Cosine similarity — measuring the angle

You don't compare embeddings by subtracting them. You measure the angle between them. Two vectors pointing in the same direction have angle ≈ 0° → cosine ≈ 1.0. Unrelated vectors point in different directions → cosine much lower.

Cosine similarity = cos(θ). Small angle → high score. Large angle → low score. Length of the vector doesn't matter.

Cosine is preferred over plain distance because embedding vectors vary wildly in magnitude. Normalising by magnitude (dividing by the norms) removes that noise.

Build it — step by step

import os

client = OpenAI(base_url="http://localhost:11434/v1", api_key="ollama")

  • The embed function

    client.embeddings.create() returns a response with a data list. Each item has an embedding field — a plain Python list[float].

    search.py
    def embed(text: str) -> list[float]:
    response = client.embeddings.create(model=EMBED_MODEL, input=text)
    return response.data[0].embedding

    🐍 Python nugget — list[float] as a type hint

    list[float] means "a list where every element is a float." Available since Python 3.9 — before that you needed from typing import List and List[float]. At runtime Python doesn't enforce this; it's for type checkers (mypy) and IDE autocomplete.

  • Cosine similarity

    Measures the angle between two vectors. Returns 1.0 (identical direction) to -1.0 (opposite). For text embeddings expect ~0.2 (unrelated) to ~0.98 (near-duplicate).

    search.py
    def cosine_similarity(a: list[float], b: list[float]) -> float:
    dot = sum(x * y for x, y in zip(a, b))
    norm_a = math.sqrt(sum(x * x for x in a))
    norm_b = math.sqrt(sum(y * y for y in b))
    return dot / (norm_a * norm_b)

    🐍 Python nugget — zip() and generator expressions

    zip(a, b) pairs elements from two iterables: zip([1,2,3], [4,5,6]) yields (1,4), (2,5), (3,6). Stops at the shorter one. Replaces index-based for i in range(len(a)) loops.

    x * y for x, y in zip(a, b) is a generator expression — like a list comprehension but lazy (doesn't allocate the full list). sum() consumes it in one pass.

  • A small document store

    Hard-code a few invoice snippets. In the next lesson (RAG) this becomes a real file store.

    search.py
    DOCS = [
    "Invoice #1001 from Acme Corp — €500 for consulting. Status: paid.",
    "Invoice #1002 from Beta Ltd — €1,200 for software licences. Status: overdue.",
    "Invoice #1003 from Gamma GmbH — €350 for hardware maintenance. Status: paid.",
    "Invoice #1004 from Delta SA — €8,400 for annual support contract. Status: overdue.",
    "Credit note #CN-01 issued to Beta Ltd for €200 — duplicate charge refund.",
]

doc_embeddings = [embed(doc) for doc in DOCS]

    🐍 Python nugget — list comprehension

    [f(x) for x in items] builds a new list by applying f to every element — more idiomatic than a for-loop with .append(). You can filter too: [f(x) for x in items if condition(x)].

    Here it embeds all documents in one line. The result is a list[list[float]] — a list of vectors.

  • The search function

    Embed the query, score every document against it, return the top k ranked by score.

    search.py
    def search(query: str, top_k: int = 3) -> list[tuple[str, float]]:
    query_vec = embed(query)
    scores = [cosine_similarity(query_vec, dv) for dv in doc_embeddings]
    ranked = sorted(zip(DOCS, scores), key=lambda pair: pair[1], reverse=True)
    return ranked[:top_k]

    🐍 Python nugget — sorted() with key=

    sorted(iterable, key=fn) sorts by the value fn(item) returns. reverse=True flips to descending. The lambda pair: pair[1] extracts the score from each (doc, score) tuple.

    Gotcha: sorted() returns a new list and never mutates the original. list.sort() is in-place and returns None. Writing result = my_list.sort() is a common bug — result will be None.

  • Run it
    search.py
    if __name__ == "__main__":
    queries = [
        "Which invoices are unpaid?",
        "Any credit notes issued?",
        "Large contracts over €5,000?",
    ]

    for query in queries:
        print(f"\nQuery: {query}")
        for doc, score in search(query, top_k=2):
            print(f"  [{score:.3f}] {doc}")

Embedding models are not interchangeable

If you embed documents with nomic-embed-text and query with text-embedding-3-small, the similarity scores are meaningless garbage. Always embed queries and documents with the same model. If you switch models, re-embed everything.
