Stateful Agents

Give your LLM a memory. Build a multi-turn assistant that understands follow-up questions.

The Problem

Every RAG call you've built so far is stateless — each question is independent. Ask "Is invoice 1004 overdue?" and it answers. Then ask "Who is the client?" and the model has no idea what "the client" refers to. Each call starts from scratch. Real assistants need to hold a conversation.

The fix is a single list. Every message — user and assistant — gets appended to a conversation history and sent on every call. The model now has context for the whole session.

How Conversation History Works

The OpenAI messages API already supports this — it accepts a list. You've been sending a list with one or two items. An agent sends the whole history:

# Turn 1
messages = [
  {"role": "system", "content": "You are an invoice assistant..."},
  {"role": "user",   "content": "Is invoice 1004 overdue?"},
]
# model replies: "Yes, it's overdue since May 15."
messages.append({"role": "assistant", "content": "Yes, it's overdue since May 15."})

# Turn 2 — send the whole list
messages.append({"role": "user", "content": "Who is the client?"})
# model now sees both turns and answers: "Delta SA, Warsaw."

The model has no memory between API calls — it only knows what's in the messages list. You are the memory.

Injecting RAG Context Per Turn

Conversation history and RAG context are two separate things. Each turn, you retrieve fresh context for the new question and inject it into the user message. The history accumulates separately.

  User types question
          │
          ▼
  embed → ChromaDB search → chunks
          │
          ▼
  Wrap: "Context:\n{chunks}\n\nQuestion: {question}"
          │
          ▼
  Append to messages list
          │
          ▼
  Send entire messages list to LLM
          │
          ▼
  Append assistant reply to messages list
          │
          ▼
  Print reply → wait for next question

The system prompt is set once. Context + question is the user message each turn. The assistant reply is appended after each call. The list grows by two entries per turn.

What You'll Build

A REPL-style invoice assistant. Run it, ask questions, get answers that reference prior turns:

You: Is invoice 1004 overdue?
Assistant: Yes — invoice #1004 for Delta SA is overdue. Due date was 2026-05-15, now 29 days late.

You: What was the original amount?
Assistant: The amount on invoice #1004 is €8,400 net for annual infrastructure support.

You: Did they request an extension?
Assistant: Yes — Delta SA requested a 30-day extension by email on 2026-05-10, but it was not agreed.

You: quit
Goodbye.

Step 2 — Write agent.py

agent.py
from pathlib import Path
from openai import OpenAI
import chromadb

client = OpenAI(base_url="http://localhost:11434/v1", api_key="ollama")
EMBED_MODEL = "nomic-embed-text"
CHAT_MODEL  = "llama3.2"

DB_DIR = Path(__file__).parent.parent / "lesson_006" / "chroma_db"
chroma = chromadb.PersistentClient(path=str(DB_DIR))
collection = chroma.get_collection(name="invoices")


def embed(text: str) -> list[float]:
    return client.embeddings.create(model=EMBED_MODEL, input=text).data[0].embedding


def retrieve(question: str) -> str:
    results = collection.query(
        query_embeddings=[embed(question)],
        n_results=3,
        include=["documents", "distances"],
    )
    chunks = [
        doc
        for doc, dist in zip(results["documents"][0], results["distances"][0])
        if dist <= 0.8
    ]
    return "\n\n".join(chunks)


def chat(messages: list[dict]) -> str:
    return (
        client.chat.completions
        .create(model=CHAT_MODEL, messages=messages, temperature=0)
        .choices[0].message.content
    )


def main():
    messages = [
        {
            "role": "system",
            "content": (
                "You are an invoice assistant. When context is provided in a user message, "
                "answer using ONLY that context. If no context is provided, use the conversation history. "
                "If you don't have enough information, say so."
            ),
        }
    ]

    print("Invoice assistant ready. Type 'quit' to exit.\n")

    while True:
        try:
            question = input("You: ").strip()
        except KeyboardInterrupt:
            print("\nGoodbye.")
            break

        if question.lower() in ("quit", "exit", "q"):
            print("Goodbye.")
            break

        if not question:
            continue

        context = retrieve(question)
        if context:
            user_content = f"Context:\n{context}\n\nQuestion: {question}"
        else:
            user_content = question

        messages.append({"role": "user", "content": user_content})
        reply = chat(messages)
        messages.append({"role": "assistant", "content": reply})

        print(f"Assistant: {reply}\n")


if __name__ == "__main__":
    main()

Python Nuggets

Python Nugget — while True / break

The standard Python pattern for a REPL loop — run forever until an explicit exit condition:

while True:
    line = input("> ")
    if line == "quit":
        break          # exits the while loop
    if not line:
        continue       # skips the rest of the loop body, goes back to top
    process(line)

break exits the nearest enclosing loop. continue skips to the next iteration. Both work in for loops too.

Python Nugget — KeyboardInterrupt

When a user presses Ctrl+C, Python raises KeyboardInterrupt. Without a handler, it prints a stack trace. Catch it to exit cleanly:

try:
    question = input("You: ")
except KeyboardInterrupt:
    print("\nGoodbye.")   # \n because Ctrl+C leaves cursor mid-line
    break

KeyboardInterrupt inherits from BaseException, not Exception — a bare except Exception won't catch it. You have to name it explicitly.

Python Nugget — str methods

A few string methods used in this lesson:

"  hello  ".strip()          # → "hello"  (removes leading/trailing whitespace)
"QUIT".lower()               # → "quit"   (lowercase)
"quit" in ("quit", "exit")  # → True    (membership test on a tuple)

The in operator works on any iterable — list, tuple, set, string. Tuples are slightly faster than lists for membership tests on small collections of constants.

The Context Window Trap

Watch out: the messages list grows by two entries every turn. After 50 turns of back-and-forth on long documents, you'll hit the model's context window limit and the call will fail. In production you'd need a strategy: summarise old turns, drop the oldest messages, or store long-term memory externally. For now, be aware the list is unbounded.

Challenges

Challenge 1 — Show turn count

Print the current turn number and how many messages are in history before each prompt. e.g. [turn 3 · 7 messages]. This makes the growing context visible.

Challenge 2 — Show retrieved context

Add a --debug flag (check sys.argv) that, when present, prints the retrieved chunks before the assistant reply. Useful for diagnosing why the model gave a wrong answer.

Challenge 3 — Sliding window memory

Keep only the system prompt plus the last N turns (e.g. 6 messages = 3 turns). Trim the list before each call: messages = [messages[0]] + messages[-6:]. Test that follow-up questions still work within the window.
