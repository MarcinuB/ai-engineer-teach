02 Your First LLM API Call

The Anthropic API follows the messages pattern: you send a list of messages (each with a role and content), and get a response back. It's a stateless HTTP call — identical semantics to any REST API you've called before.

Create first_call.py:

first_call.py 
import anthropic
from dotenv import load_dotenv

load_dotenv()  # loads .env file into environment variables

# Client reads ANTHROPIC_API_KEY from env automatically
client = anthropic.Anthropic()

response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    messages=[
        {"role": "user", "content": "What is a context window in LLMs?"}
    ]
)

# response.content is a list of content blocks
print(response.content[0].text)

# Always check your token usage
print(f"\nTokens used: {response.usage.input_tokens} in / {response.usage.output_tokens} out")
(.venv) $ python first_call.py
A context window in large language models refers to the maximum amount of text (measured in tokens) that the model can process in a single interaction... Tokens used: 18 in / 187 out

Notice the response object structure. response.content is a list because the API can return multiple content blocks (text, tool calls, images). For a simple text response, you always want content[0].text.

The context window is the total token budget for one API call — input tokens + output tokens combined. Claude Sonnet 4.6 has a 200k token input limit. max_tokens caps the output size specifically.

03 Structured Output with Pydantic

Free-form text responses are fine for chat. But production systems need reliable, typed data. You can't json.parse() a prose response.

The pattern:

  1. Define a Pydantic model describing the shape you want
  2. Tell Claude in the system prompt to return only JSON matching that shape
  3. Parse and validate the response with Pydantic — it raises a clean exception if the model returns bad data

Create extractor.py:

extractor.py 
import json
import anthropic
from dotenv import load_dotenv
from pydantic import BaseModel, ValidationError
from typing import Optional

load_dotenv()
client = anthropic.Anthropic()


# 1. Define the exact shape you want back
class Invoice(BaseModel):
    vendor: str
    amount: float
    date: str
    description: Optional[str] = None


# 2. Call Claude with a tight system prompt
def extract_invoice(raw_text: str) -> Invoice:
    response = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=512,
        system="""You are a data extraction engine.
Extract invoice data from the user's text and return ONLY a JSON object.
No prose, no explanation, no markdown code fences — just the raw JSON.
Required fields: vendor (string), amount (float), date (string), description (string or null).""",
        messages=[
            {"role": "user", "content": raw_text}
        ]
    )

    raw_json = response.content[0].text

    # 3. Parse JSON, then validate with Pydantic
    try:
        data = json.loads(raw_json)
        return Invoice(**data)
    except json.JSONDecodeError:
        raise ValueError(f"Model returned invalid JSON: {raw_json[:200]}")
    except ValidationError as e:
        raise ValueError(f"Model returned wrong shape: {e}")


# ---- Test it ----
sample_text = """
Invoice received from Acme Software Ltd, dated March 15, 2024.
Please process payment of $3,750.00 for Q1 consulting services
covering API integration and system architecture review.
"""

invoice = extract_invoice(sample_text)

print(f"Vendor:      {invoice.vendor}")
print(f"Amount:      ${invoice.amount:,.2f}")
print(f"Date:        {invoice.date}")
print(f"Description: {invoice.description}")
print()
print("As dict:", invoice.model_dump())
(.venv) $ python extractor.py
Vendor: Acme Software Ltd Amount: $3,750.00 Date: March 15, 2024 Description: Q1 consulting services covering API integration and system architecture review As dict: {'vendor': 'Acme Software Ltd', 'amount': 3750.0, 'date': 'March 15, 2024', 'description': 'Q1 consulting services...'}

Why Pydantic instead of just json.loads()?

json.loads() gives you a raw dict — you still don't know if amount is a float, a string, or missing. Pydantic enforces your schema at runtime: wrong type → coerce if possible, missing required field → raise immediately. This turns silent data bugs into loud exceptions.

System prompts are standing orders. "Return ONLY JSON" in the system prompt dramatically reduces the chance the model adds prose around your JSON. Still validate — models occasionally add markdown code fences (```json ... ```). Strip those before parsing if needed.
