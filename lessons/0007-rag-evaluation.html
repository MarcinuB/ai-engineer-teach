RAG Evaluation

You have a working RAG pipeline. Now measure whether it's actually any good.

The Problem

You've been testing your RAG by running a question and reading the answer. That's fine for development — but it doesn't scale. As soon as you change your chunk size, your similarity threshold, your system prompt, or your model, you need to know: did things get better or worse? Without a metric, you're flying blind.

This lesson teaches you to build an eval harness — a repeatable, automated test suite that scores your RAG pipeline on a set of known questions. You'll use the LLM-as-judge pattern, where another LLM call scores each answer.

The Two Metrics That Matter

There are many RAG evaluation frameworks (RAGAS, TruLens, etc.), but they all trace back to the same two fundamental questions:

Faithfulness

Is the answer grounded in the retrieved context, or is the model hallucinating?

Answer Relevance

Does the answer actually address what the user asked?

Context Relevance

Did retrieval fetch the right chunks? (Left as a challenge below.)

Today you'll implement the first two. A production RAG needs all three — but faithfulness + relevance will already catch most real failures.

LLM-as-Judge

The trick is using an LLM to score LLM outputs. This sounds circular, but it works in practice for two reasons:

  • The judge sees the context and the answer simultaneously — it can compare them in a way you can't easily automate with string matching.
  • It produces a numeric score, which you can track over time and average across a test set.
Production note: LLM-as-judge adds latency and cost — each test case requires extra model calls. In production, run evals as an offline batch job, not on every request. Think of it like a test suite: you run it before deploying changes, not per-user.

Step 2 — Write eval.py

eval.py
from dataclasses import dataclass
import json
from pathlib import Path
from openai import OpenAI
import chromadb

client = OpenAI(base_url="http://localhost:11434/v1", api_key="ollama")
EMBED_MODEL = "nomic-embed-text"
CHAT_MODEL  = "llama3.2"

DB_DIR = Path(__file__).parent.parent / "lesson_006" / "chroma_db"
chroma = chromadb.PersistentClient(path=str(DB_DIR))
collection = chroma.get_collection(name="invoices")


# ── Test cases ────────────────────────────────────────────────────────────────
@dataclass
class TestCase:
    question: str
    must_contain: str  # key fact the answer should include; empty = expect refusal


TEST_CASES = [
    TestCase("What is the amount on invoice 1002?",     "1,200"),
    TestCase("Who is the client on invoice 1004?",       "Delta"),
    TestCase("What is the due date for invoice 1002?",   "2026-05-31"),
    TestCase("Is invoice 1004 overdue?",                 "overdue"),
    TestCase("What is credit note CN-01 for?",            "duplicate"),
    TestCase("What is the weather in Warsaw?",            ""),
]


# ── RAG ───────────────────────────────────────────────────────────────────────
def embed(text: str) -> list[float]:
    return client.embeddings.create(model=EMBED_MODEL, input=text).data[0].embedding


def rag(question: str) -> tuple[str, str]:
    results = collection.query(
        query_embeddings=[embed(question)],
        n_results=3,
        include=["documents", "distances"],
    )
    chunks = [
        doc
        for doc, dist in zip(results["documents"][0], results["distances"][0])
        if dist < 0.5
    ]
    context = "\n\n".join(chunks)

    if not context:
        return "", "I don't have relevant information to answer that."

    messages = [
        {"role": "system", "content": f"Answer using ONLY this context:\n\n{context}"},
        {"role": "user",   "content": question},
    ]
    answer = client.chat.completions.create(
        model=CHAT_MODEL, messages=messages
    ).choices[0].message.content
    return context, answer


# ── LLM-as-judge ─────────────────────────────────────────────────────────────
def judge(prompt: str) -> tuple[int, str]:
    resp = client.chat.completions.create(
        model=CHAT_MODEL,
        messages=[{"role": "user", "content": prompt}],
        response_format={"type": "json_object"},
    )
    data = json.loads(resp.choices[0].message.content)
    return int(data["score"]), data["reason"]


def score_faithfulness(context: str, answer: str) -> tuple[int, str]:
    return judge(f"""Rate how faithful this answer is to the context.
5 = fully supported, no hallucinations. 1 = contradicts or ignores context.
Return JSON: {{"score": <1-5>, "reason": "<one sentence>"}}

Context: {context}
Answer: {answer}""")


def score_relevance(question: str, answer: str, must_contain: str) -> tuple[int, str]:
    hint = (
        f"The answer should mention: {must_contain}"
        if must_contain
        else "A correct answer here says it has no relevant information."
    )
    return judge(f"""Rate how well this answer addresses the question.
5 = directly and completely answers it. 1 = misses the point entirely.
{hint}
Return JSON: {{"score": <1-5>, "reason": "<one sentence>"}}

Question: {question}
Answer: {answer}""")


# ── Main ──────────────────────────────────────────────────────────────────────
PASS_THRESHOLD = 3

if __name__ == "__main__":
    passed = 0
    print(f"\n{'#':3} {'Question':42} {'Faith':>5} {'Relev':>5} {'OK?':>4}")
    print("─" * 65)

    for i, tc in enumerate(TEST_CASES, 1):
        context, answer = rag(tc.question)

        if context:
            f_score, _ = score_faithfulness(context, answer)
        else:
            f_score = 5  # refusal is always faithful — nothing was hallucinated

        r_score, r_reason = score_relevance(tc.question, answer, tc.must_contain)
        ok = f_score >= PASS_THRESHOLD and r_score >= PASS_THRESHOLD
        if ok:
            passed += 1

        q = tc.question[:39] + "..." if len(tc.question) > 39 else tc.question
        print(f"{i:<3} {q:<42} {f_score:>5} {r_score:>5} {'✓' if ok else '✗':>4}")
        print(f"    A: {answer[:90]}")
        if not ok:
            print(f"    ↳ {r_reason}")
        print()

    rate = passed / len(TEST_CASES) * 100
    print(f"Pass rate: {passed}/{len(TEST_CASES)} ({rate:.0f}%)")

Python Nuggets

Python Nugget — @dataclass

@dataclass auto-generates __init__, __repr__, and __eq__ for you. Think of it as Python's equivalent of a TypeScript interface with values, or a Java POJO without the ceremony:

@dataclass
class TestCase:
    question: str
    must_contain: str

# Equivalent hand-written version (don't do this):
class TestCase:
    def __init__(self, question: str, must_contain: str):
        self.question = question
        self.must_contain = must_contain

Dataclasses are the go-to for simple structured data in Python. Use @dataclass(frozen=True) for immutable instances.

Python Nugget — f-string alignment

F-strings support format specs for padding — useful for tabular output:

name = "invoice 1002?"
score = 5

print(f"{name:<42}")   # left-aligned in 42 chars:  "invoice 1002?                             "
print(f"{score:>5}")   # right-aligned in 5 chars:  "    5"
print(f"{score:^5}")   # centered in 5 chars:       "  5  "
print(f"{0.667:.0f}")  # float, 0 decimal places:   "1"
print(f"{0.667:.1%}")  # percentage, 1 decimal:     "66.7%"

Align chars: < left, > right, ^ center. Format: {value:[align][width][.precision][type]}

Python Nugget — json.loads()

When an API returns a JSON string, json.loads() parses it to a Python dict/list. Ollama's JSON mode gives you the model's output as a JSON string inside the response content — not auto-parsed:

import json

raw = '{"score": 4, "reason": "Answer is supported by context"}'
data = json.loads(raw)       # → dict
data["score"]               # → 4 (int, not string)

# json.dumps() goes the other way:
json.dumps(data)             # → '{"score": 4, "reason": "..."}'
json.dumps(data, indent=2)  # pretty-printed

How LLM-as-Judge Works

The judge() function asks the model to return structured JSON. The key is response_format={"type": "json_object"} — this tells Ollama to force JSON output, preventing the model from wrapping its response in markdown or prose.

Your prompt must describe the JSON schema you want. The model won't infer it — you have to tell it: "Return JSON: {"score": <1-5>, "reason": "<one sentence>"}"

  Test case
  ─────────
  question: "What is the amount on invoice 1002?"
  must_contain: "1,200"
          │
          ▼
  ┌─────────────────────┐
  │   rag(question)     │
  │  embed → search     │
  │  → inject → LLM     │
  └──────────┬──────────┘
             │
       context, answer
       /            \
      ▼              ▼
  ┌──────────┐  ┌──────────────────────┐
  │ faithful-│  │ relevance judge       │
  │ ness     │  │ (question vs answer) │
  │ judge    │  └──────────┬───────────┘
  └────┬─────┘             │
       │ f_score           │ r_score
       └─────────┬─────────┘
                 ▼
            ok = both >= 3

Challenge 2 — Add test cases

Challenge 3 — Context relevance metric

Add a third metric: context relevance — did the retrieval actually fetch useful chunks? Write a score_context_relevance(question, context) judge that scores whether the context contains information relevant to the question (5 = directly relevant, 1 = completely off-topic). This is the third leg of the RAG eval triad. Together, all three let you debug where failures happen: retrieval, generation, or grounding.
