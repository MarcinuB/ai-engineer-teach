AI Engineering Lesson 03

Tool Calling

Give the LLM hands — it decides when to call your Python functions.

What is tool calling?

So far the LLM has only returned text. Tool calling flips this: the LLM can request to run code you wrote, you execute it, return the result, and the LLM uses that result to produce its final answer.

You define the tools. The LLM decides if and when to call them.

The loop keeps running until finish_reason == "stop". The LLM can call zero, one, or many tools per turn.

Step 0 — start Ollama

Ollama exposes an OpenAI-compatible HTTP API at localhost:11434. You need the server running before your script makes any calls.

Start the server

ollama serve

Leave this running in a terminal tab. If Ollama is already running as a system service, this command will fail with "address already in use" — that's fine, the API is already up.

Pull a model that supports tool calling

ollama pull llama3.2

Which Ollama models support tool calling?

llama3.2, llama3.1, qwen2.5, mistral-nemo. Avoid very small models (<3B) — they often produce malformed JSON for tool calls.

Verify it's running

curl http://localhost:11434/v1/models

You should get back a JSON list of available models.

Provider landscape

The OpenAI SDK is the de-facto standard interface — most providers implement a compatible HTTP endpoint so one client covers everything.

Local Ollama

Free, private, no API key.
base_url = "http://localhost:11434/v1"
api_key = "ollama" (any string)

Cloud OpenAI

Default SDK endpoint.
api_key = OPENAI_API_KEY
Good cheap option: gpt-4o-mini

Cloud Anthropic

OpenAI-compatible endpoint.
base_url = "https://api.anthropic.com/v1/"
api_key = ANTHROPIC_API_KEY

Same Python code, different constructor args — only the base_url and api_key differ.

Config — .env in the parent directory

Your scripts live in lesson_003/ but the .env is one level up at invoice-extractor/.env. You need to tell python-dotenv to look for it by path, not by the default "search upward from cwd" behaviour: 

from pathlib import Path
from dotenv import load_dotenv

# __file__ is this script; .parent is lesson_003/; .parent again is invoice-extractor/
load_dotenv(Path(__file__).parent.parent / ".env")

Ollama needs no key at all — "ollama" is a placeholder that satisfies the SDK. Only add load_dotenv when you switch to a cloud provider.

Build the script — step by step

Create invoice-extractor/lesson_003/tool_call.py and add each piece below in order. Read each block before you type it — understand what it does, then add it.

  • Imports and client

    Install the SDK first if needed: pip install openai

    tool_call.py
    import json
import os
from openai import OpenAI

client = OpenAI(
    base_url="http://localhost:11434/v1",
    api_key="ollama",
)
MODEL = "llama3.2"

    Swap these two lines for any other provider and the rest of your code stays the same.

  • Define a tool

    A tool definition is a dict with three things: a name (called by the model), a description (read by the model to decide when to use it), and a parameters JSON Schema (tells the model what arguments to supply).

    tool_call.py
    VAT_RATES = {"PL": 0.23, "DE": 0.19, "FR": 0.20, "GB": 0.20, "US": 0.0}

def get_vat_rate(country_code: str) -> dict:
    rate = VAT_RATES.get(country_code.upper())
    if rate is None:
        return {"error": f"Unknown country: {country_code}"}
    return {"country_code": country_code.upper(), "vat_rate": rate}


tools = [
    {
        "type": "function",
        "function": {
            "name": "get_vat_rate",
            "description": "Returns the VAT rate for a country as a decimal, e.g. 0.23 for Poland.",
            "parameters": {
                "type": "object",
                "properties": {
                    "country_code": {
                        "type": "string",
                        "description": "ISO 3166-1 alpha-2 code, e.g. 'PL'",
                    }
                },
                "required": ["country_code"],
            },
        },
    }
]

    The description is the only thing that tells the LLM when to call this. Vague descriptions = wrong calls or no calls. Be specific.

  • The tool dispatch map

    You'll have one or more tool functions. A dict mapping name → function keeps the dispatch loop clean — no if/elif chains.

    tool_call.py
    TOOL_MAP = {
    "get_vat_rate": get_vat_rate,
}

def run_tool(name: str, arguments: dict):
    fn = TOOL_MAP.get(name)
    if not fn:
        return {"error": f"Unknown tool: {name}"}
    return fn(**arguments)
  • The loop

    This is the core pattern. Keep calling the model until it stops requesting tools. Each iteration: check finish_reason, execute any tool calls, append results, repeat.

    tool_call.py
    def run_with_tools(user_message: str) -> str:
    messages = [{"role": "user", "content": user_message}]

    while True:
        response = client.chat.completions.create(
            model=MODEL,
            messages=messages,
            tools=tools,
        )
        choice = response.choices[0]

        if choice.finish_reason == "stop":
            return choice.message.content

        if choice.finish_reason == "tool_calls":
            messages.append(choice.message)   # assistant's request goes into history

            for tc in choice.message.tool_calls:
                args = json.loads(tc.function.arguments)
                result = run_tool(tc.function.name, args)
                print(f"  [tool] {tc.function.name}({args}) → {result}")

                messages.append({
                    "role": "tool",
                    "tool_call_id": tc.id,
                    "content": json.dumps(result),
                })
            # loop: model now sees the tool results and continues

    Don't forget to append tool results

    If you run the function but don't append the "tool" message back into messages, the model loops forever waiting for a result that never arrives. The result must be in the conversation history.

  • Run it
    tool_call.py
    if __name__ == "__main__":

Challenge — add a second tool

Once the basic loop works, extend it. Add a convert_currency tool that takes amount, from_currency, to_currency and returns a converted value using hardcoded rates (EUR→PLN ≈ 4.25 is fine for now).

Then ask: "I'm invoicing a Polish client for €1,200 net. What's the gross total in PLN?"

The model should call both tools — first get_vat_rate, then convert_currency. Watch the [tool] log lines to verify the sequence.

Tip — Pydantic can generate JSON schema for you

Writing JSON schema by hand gets tedious. Define the tool's input as a Pydantic model and call model.model_json_schema() to get the parameters dict automatically. Ask your agent to show you the pattern.
