Your extractor from Lesson 01 works — in a happy-path test. In production, you'll face rate limits, transient server errors, and a model that occasionally wraps its JSON in markdown. None of these are bugs in your code. They're expected failure modes you need to handle explicitly.

By the end of this lesson, your extractor will handle all of them gracefully.

01 A Map of What Can Go Wrong

LLM API calls have failure modes at two layers: HTTP/network (standard API failures) and LLM output (the model misbehaved). You handle them differently.

FailureLayerCauseFix
429 Rate limit HTTP Too many requests per minute Exponential backoff + retry
529 Overloaded HTTP Provider under heavy load Backoff + retry
5xx Server error HTTP Transient provider failure Backoff + retry (max 3x)
4xx Client error HTTP Bad request (wrong model name, exceeded context) Fix your code — do not retry
JSON fence LLM output Model adds ```json … ``` around output Strip with regex before parsing
Invalid JSON LLM output Model truncated (hit max_tokens) or hallucinated syntax Send correction message + retry call
Wrong schema LLM output Missing field, wrong type (e.g. amount as string) Pydantic catches it → send correction + retry
The golden rule Retry on 429, 529, 5xx, and bad LLM output. Never retry on 4xx — those mean your request is wrong, not the server. Retrying a 400 just hammers the API with the same broken request.

02 Handling HTTP Errors with Backoff

The Anthropic SDK raises typed exceptions — you don't parse status codes yourself. The key ones:

exception types (Anthropic SDK)
anthropic.RateLimitError      # 429 — back off and retry
anthropic.APIStatusError      # all other HTTP errors — check .status_code
anthropic.APIConnectionError  # network failure — retry

Wrap your API call in a retry loop with exponential backoff — wait 1s after the first failure, 2s after the second, 4s after the third. This prevents hammering the API while it recovers.

retry.py — HTTP-level retry helper 
import time
import anthropic

def create_with_retry(client: anthropic.Anthropic, max_retries: int = 3, **kwargs):
    """Drop-in wrapper around client.messages.create with retry logic."""
    for attempt in range(max_retries):
        try:
            return client.messages.create(**kwargs)

        except (anthropic.RateLimitError, anthropic.APIConnectionError) as e:
            if attempt == max_retries - 1:
                raise
            wait = 2 ** attempt          # 1s, 2s, 4s
            print(f"[retry] {type(e).__name__}, waiting {wait}s (attempt {attempt+1}/{max_retries})")
            time.sleep(wait)

        except anthropic.APIStatusError as e:
            if e.status_code < 500:
                raise                       # 4xx → your fault, don't retry
            if attempt == max_retries - 1:
                raise
            wait = 2 ** attempt
            print(f"[retry] {e.status_code} server error, waiting {wait}s")
            time.sleep(wait)
Why **kwargs? It forwards any keyword arguments through unchanged — so create_with_retry(client, model=..., messages=...) works identically to client.messages.create(model=..., messages=...). This is a common Python pattern for transparent wrappers.

03 Stripping Markdown Fences

Even with "return ONLY JSON" in your system prompt, Claude occasionally returns:

what the model sometimes returns
```json
{"vendor": "Acme", "amount": 3750.0, "date": null}
```

That's valid JSON wrapped in a markdown code fence — json.loads() will fail on it. Strip the fence with a simple regex before parsing:

json_utils.py 
import re

def strip_json_fence(text: str) -> str:
    """Remove ```json ... ``` or ``` ... ``` wrappers if present."""
    text = text.strip()
    # Match optional language tag: ```json or just ```
    match = re.search(r'```(?:json)?\s*([\s\S]*?)```', text)
    if match:
        return match.group(1).strip()
    return text
>>> strip_json_fence('```json\n{"vendor": "Acme"}\n```')
'{"vendor": "Acme"}'
>>> strip_json_fence('{"vendor": "Acme"}')
'{"vendor": "Acme"}'

This is idempotent — if there's no fence, it returns the text unchanged. Always run it before json.loads().

04 LLM Self-Correction

When the model returns unparseable or schema-invalid JSON, don't just raise. Tell the model what it did wrong and ask it to fix it. Models are surprisingly good at correcting their own output when you show them the error.

This works by continuing the conversation: append the bad response as an assistant message, then add a user message explaining the problem.

self-correction pattern 
import json
import anthropic
from pydantic import BaseModel, ValidationError
from json_utils import strip_json_fence

def extract_with_correction(
    client: anthropic.Anthropic,
    model_class: type[BaseModel],
    messages: list,
    system: str,
    model: str = "claude-sonnet-4-6",
    max_corrections: int = 2,
) -> BaseModel:
    """Call the LLM and retry with correction message if output is invalid."""
    conversation = list(messages)   # don't mutate caller's list

    for attempt in range(max_corrections + 1):
        response = client.messages.create(
            model=model,
            max_tokens=1024,
            system=system,
            messages=conversation,
        )
        raw = strip_json_fence(response.content[0].text)

        try:
            data = json.loads(raw)
            return model_class(**data)

        except (json.JSONDecodeError, ValidationError) as e:
            if attempt == max_corrections:
                raise ValueError(f"Model failed after {max_corrections} corrections: {e}")

            # Append bad response + correction request to conversation
            conversation.append({"role": "assistant", "content": response.content[0].text})
            conversation.append({
                "role": "user",
                "content": (
                    f"Your response was invalid. Error: {e}\n\n"
                    f"Return ONLY valid JSON with no prose or fences. Try again."
                )
            })
            print(f"[correction] attempt {attempt+1}: {type(e).__name__}")
Why does appending to the conversation work? The LLM has no memory between calls — each call is stateless. When you append the previous bad response as an assistant message, you're literally showing the model what it said and asking it to fix it. It can see its own mistake and correct it.

05 Putting It All Together

Here's the production-grade version of your invoice extractor, combining all three layers:

extractor_v2.py — complete production extractor 
import json
import re
import time
import anthropic
from dotenv import load_dotenv
from pydantic import BaseModel, ValidationError
from typing import Optional

load_dotenv()
client = anthropic.Anthropic()


# ── Models ──────────────────────────────────────────────────────────────────

class LineItem(BaseModel):
    description: str
    amount: float

class Invoice(BaseModel):
    vendor: str
    amount: float
    date: Optional[str] = None
    description: Optional[str] = None
    line_items: list[LineItem] = []


# ── Helpers ──────────────────────────────────────────────────────────────────

def strip_fence(text: str) -> str:
    text = text.strip()
    match = re.search(r'```(?:json)?\s*([\s\S]*?)```', text)
    return match.group(1).strip() if match else text

def create_with_retry(max_retries=3, **kwargs):
    for attempt in range(max_retries):
        try:
            return client.messages.create(**kwargs)
        except (anthropic.RateLimitError, anthropic.APIConnectionError) as e:
            if attempt == max_retries - 1: raise
            time.sleep(2 ** attempt)
        except anthropic.APIStatusError as e:
            if e.status_code < 500: raise
            if attempt == max_retries - 1: raise
            time.sleep(2 ** attempt)


SYSTEM = """You are a data extraction engine.
Extract invoice data and return ONLY a JSON object — no prose, no fences.
Fields: vendor (str), amount (float), date (str|null), description (str|null),
line_items (array of {description: str, amount: float})."""


# ── Main extractor ───────────────────────────────────────────────────────────

def extract_invoice(raw_text: str, max_corrections: int = 2) -> Invoice:
    conversation = [{"role": "user", "content": raw_text}]

    for attempt in range(max_corrections + 1):
        response = create_with_retry(
            model="claude-sonnet-4-6",
            max_tokens=1024,
            system=SYSTEM,
            messages=conversation,
        )
        raw = strip_fence(response.content[0].text)

        try:
            return Invoice(**json.loads(raw))
        except (json.JSONDecodeError, ValidationError) as e:
            if attempt == max_corrections:
                raise ValueError(f"Failed after {max_corrections} corrections: {e}")
            conversation += [
                {"role": "assistant", "content": response.content[0].text},
                {"role": "user", "content": f"Invalid response — {e}. Return only valid JSON."},
            ]
            print(f"[correction {attempt+1}] {type(e).__name__}")


# ── Test ──────────────────────────────────────────────────────────────────────

sample = """OVERDUE NOTICE — Invoice #INV-2024-089
From: DataFlow Solutions (UK)
Total: £12,400.00 GBP — PAYMENT 30 DAYS OVERDUE

Items:
- Cloud infrastructure audit: £4,500
- Security penetration testing: £5,200
- Remediation report and recommendations: £2,700
"""

invoice = extract_invoice(sample)
print(f"Vendor:     {invoice.vendor}")
print(f"Amount:     {invoice.amount}")
print(f"Date:       {invoice.date}")
print(f"Line items: {len(invoice.line_items)}")
for item in invoice.line_items:
    print(f"  - {item.description}: {item.amount}")
Notice the structure: extract_invoice owns the retry-on-bad-output loop. create_with_retry owns the HTTP-level retry loop. These are two separate concerns — keep them separate. One handles network failures, the other handles LLM output failures.
