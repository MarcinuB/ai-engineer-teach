Lesson 06 · Days 31–60: Production RAG

Chunking & Vector Stores

Your RAG pipeline from lesson 05 works — but it has two problems that will bite you on any real project. This lesson fixes both.

The two problems with lesson 05

1. You re-embed every time you run the script. Embedding 100 documents takes a second or two. Embedding 10,000 takes minutes — and you'd be doing it on every startup.

2. You embed whole documents. If a document is 3,000 words, the embedding averages everything together. "Invoice 001 is overdue" and "Invoice 002 was paid" collapse into one blurry vector. Retrieval loses precision, and you may inject far more text than the model's context window can handle.

The industry solution: chunk first, then store embeddings in a vector database that persists to disk. You only embed once; after that it's just a query.

Chunking

A chunk is a short, self-contained excerpt of a document — usually 200–500 words. The goal is to make each chunk answer a single question well.

Overlap

If you split rigidly at chunk boundaries, a sentence can get cut in half. Overlap means each chunk shares some words with the next one, so no context falls into a gap.

Example: chunk size 300 words, overlap 50 words. Chunk 0 = words 0–299. Chunk 1 = words 250–549. The 50-word overlap is shared between both.

ChromaDB

ChromaDB is a local vector database written in Python. It:

  • Stores embeddings + metadata + original text together
  • Persists to disk — no re-embedding on restart
  • Queries by semantic similarity in one call
  • Runs fully local, no API key needed
OpenAI alternative: In production you'd use a hosted store like Pinecone, Weaviate, or pgvector. ChromaDB is the right choice for local development — same concepts, zero ops.

What You'll Build

A two-file pipeline: ingest.py (chunk + embed + store, run once) and rag.py (query + answer, run any time).

Step 2 — Write the chunker

Create invoice-extractor/lesson_006/ingest.py. Start with a chunk function:

def chunk_text(text: str, size: int = 300, overlap: int = 50) -> list[str]:
    words = text.split()
    chunks = []
    start = 0
    while start < len(words):
        end = start + size
        chunk = " ".join(words[start:end])
        chunks.append(chunk)
        if end >= len(words):
            break
        start += size - overlap
    return chunks

This splits on whitespace, slides a window of size words, stepping size - overlap each time.

Python nugget — list slicing

words[start:end] returns a new list from index start up to (not including) end. No error if end goes past the list length — Python just returns what's there.

[10, 20, 30, 40][1:10]  # → [20, 30, 40] — no IndexError

Contrast with indexing: words[10] on a 4-element list throws IndexError, but words[1:10] does not.

Step 3 — Set up ChromaDB + embed client

Add imports and initialise everything at the top of ingest.py:

import chromadb
from pathlib import Path
from openai import OpenAI

client = OpenAI(base_url="http://localhost:11434/v1", api_key="ollama")
EMBED_MODEL = "nomic-embed-text"

DOCS_DIR = Path(__file__).parent / "docs"
DB_DIR   = Path(__file__).parent / "chroma_db"

chroma = chromadb.PersistentClient(path=str(DB_DIR))
collection = chroma.get_or_create_collection(name="invoices")

PersistentClient saves to disk. get_or_create_collection is idempotent — safe to call every run.

Python nugget — str() on Path

Many older libraries (including ChromaDB) expect a plain str path, not a pathlib.Path object. Wrap with str() when passing to them:

p = Path("/some/dir")
open(p)        # fine — built-ins accept Path
some_lib(p)   # might crash — use str(p) instead
Step 4 — Chunk, embed, and store

Add an embed helper, then the ingestion loop:

def embed(text: str) -> list[float]:
    resp = client.embeddings.create(model=EMBED_MODEL, input=text)
    return resp.data[0].embedding


def ingest():
    for path in sorted(DOCS_DIR.glob("*.txt")):
        text = path.read_text(encoding="utf-8")
        chunks = chunk_text(text)

        for i, chunk in enumerate(chunks):
            doc_id = f"{path.stem}_chunk{i}"
            collection.add(
                ids=[doc_id],
                documents=[chunk],
                embeddings=[embed(chunk)],
                metadatas=[{"source": path.name, "chunk": i}],
            )
            print(f"  stored {doc_id}")

        print(f"Ingested {path.name}: {len(chunks)} chunks")


if __name__ == "__main__":
    ingest()

Each chunk gets a unique ID (invoice_001_chunk0), its embedding, and metadata so you know which file and chunk index it came from.

Run it:

python invoice-extractor/lesson_006/ingest.py

You should see one line per chunk. The chroma_db/ folder will appear on disk.

Python nugget — f-string with attribute access

path.stem gives the filename without extension. You can call attributes and methods inside f"..." curly braces:

p = Path("invoice_001.txt")
f"{p.stem}"   # → "invoice_001"
f"{p.name}"   # → "invoice_001.txt"
f"{p.suffix}" # → ".txt"
Step 5 — Query from ChromaDB

Create invoice-extractor/lesson_006/rag.py. This file doesn't embed documents — it only queries:

import chromadb
from pathlib import Path
from openai import OpenAI

client = OpenAI(base_url="http://localhost:11434/v1", api_key="ollama")
EMBED_MODEL = "nomic-embed-text"
CHAT_MODEL  = "llama3.2"

DB_DIR = Path(__file__).parent / "chroma_db"
chroma = chromadb.PersistentClient(path=str(DB_DIR))
collection = chroma.get_collection(name="invoices")


def embed(text: str) -> list[float]:
    resp = client.embeddings.create(model=EMBED_MODEL, input=text)
    return resp.data[0].embedding


def search(query: str, top_k: int = 3, max_distance: float = 0.5) -> list[dict]:
    results = collection.query(
        query_embeddings=[embed(query)],
        n_results=top_k,
        include=["documents", "metadatas", "distances"],
    )
    chunks = results["documents"][0]
    metas  = results["metadatas"][0]
    dists  = results["distances"][0]
    return [
        {"text": c, "source": m["source"], "distance": d}
        for c, m, d in zip(chunks, metas, dists)
        if d <= max_distance
    ]


def rag_query(question: str) -> str:
    results = search(question)

    for i, r in enumerate(results):
        print(f"  [{i+1}] dist={r['distance']:.3f} src={r['source']} | {r['text'][:80]}")

    context = "\n\n".join(
        f"[Source: {r['source']}]\n{r['text']}" for r in results
    )

    system_prompt = f"""You are an invoice assistant. Answer using ONLY the documents below.
If the answer is not in the documents, say so explicitly.

Documents:
{context}"""

    resp = client.chat.completions.create(
        model=CHAT_MODEL,
        messages=[
            {"role": "system", "content": system_prompt},
            {"role": "user", "content": question},
        ],
    )
    return resp.choices[0].message.content


if __name__ == "__main__":
    questions = [
        "Which invoices are overdue?",
        "Did Beta Ltd receive any credit notes?",
        "What is the weather in Warsaw today?",
    ]
    for q in questions:
        print(f"\nQ: {q}")
        print(f"A: {rag_query(q)}")

Notice that ChromaDB uses distance, not similarity. Lower distance = more similar. It uses L2 (Euclidean) distance by default — opposite direction from cosine similarity score.

Distance vs Similarity

Your lesson 05 code used cosine similarity (higher = more similar, max 1.0). ChromaDB returns distance (lower = more similar, min 0.0). To filter by distance, you'd use distance < 0.8 instead of score > 0.6.

You can switch ChromaDB to cosine by passing metadata={"hnsw:space": "cosine"} to get_or_create_collection — but for now the default is fine. Just remember: distance, not score.

Challenges

Challenge 1 — Cosine space

Change get_or_create_collection in ingest.py to use cosine distance:

collection = chroma.get_or_create_collection(
    name="invoices",
    metadata={"hnsw:space": "cosine"},
)

Delete chroma_db/ and re-run ingest.py. Then update your threshold in rag.py — distances are now 0.0–1.0 where 0.0 = identical. What threshold makes sense?

Challenge 2 — Show sources in the answer

Change rag_query to return both the answer and the list of source files that were used. Print them separately:

A: Invoice 001 and Invoice 003 are overdue.
Sources: invoice_001.txt, invoice_003.txt

Hint: the metadata dict has a "source" key. You already have results in scope.

Challenge 3 — Idempotent ingest

Right now, running ingest.py twice will try to add duplicate IDs and crash. Fix it: before adding chunks for a file, check if that file's chunks already exist and skip it.

Hint: collection.get(ids=[doc_id]) returns a result with an "ids" list — check if it's empty.
